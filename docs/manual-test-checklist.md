## 一、全新用户流程

### 测试场景

验证没有安装 OpenClaw、没有 OpenClaw 配置和安装器状态时的首次使用入口。

### 准备状态

- `openclaw` 命令不存在。
- `~/.openclaw` 不存在。
- `~/.openclaw-installer` 不存在。
- 可使用文末“删除 OpenClaw”命令准备测试环境。

### 操作步骤

1. 正常启动 GUI。
2. 检查首页标题和说明。
3. 检查首页最醒目的操作按钮。
4. 检查是否出现完整仪表盘。
5. 点击“开始准备 OpenClaw”。
6. 检查当前步骤和向导步骤条。

### 预期结果

- 首页显示“欢迎使用 OpenClaw 工具箱”。
- 页面只突出“开始准备 OpenClaw”这一项主操作。
- 不显示已配置用户使用的完整仪表盘。
- 点击后进入首次向导第 1 步“准备 OpenClaw”。
- 向导显示四步：准备 OpenClaw、配置 API Key、检查配置、打开控制台。
- 不直接进入配置 API Key 页面，也不跳到第 2 步。

### 备注/异常处理

- 如果首页误判为已安装，运行 `which openclaw`，确认系统中是否还有同名命令。
- 如果仍显示旧配置状态，确认 `~/.openclaw-installer` 已删除后重新启动 GUI。

## 二、未安装 OpenClaw 时点击配置 API Key

### 测试场景

验证 OpenClaw 未安装时，配置入口不会提前展示 API Key 表单。

### 准备状态

- 保持全新用户的未安装状态。
- GUI 位于首页。

### 操作步骤

1. 点击左侧“配置 API Key”。
2. 检查页面标题、说明和按钮。
3. 确认页面中没有 API Key 输入框。
4. 点击“去准备 OpenClaw”。
5. 返回提示页后点击“重新检查”。
6. 再次返回提示页，点击“打开问题排查”。

### 预期结果

- 页面显示“需要先准备 OpenClaw”。
- 不显示 AI 服务商、API Key、模型等配置表单。
- “去准备 OpenClaw”进入首次向导第 1 步，不进入第 2 步。
- “重新检查”有检查中或检查完成反馈，页面不会无响应。
- “打开问题排查”进入问题排查页面。

### 备注/异常处理

- 如果直接出现 API Key 表单，记录首页状态、`which openclaw` 输出和复现步骤。
- 重新检查不应自动打开基础组件准备页。

## 三、Node.js、npm、Git 模拟缺失

### 测试场景

验证开发专用缺失依赖模拟只影响诊断，不阻断普通用户主流程。

### 准备状态

- 确保通过非打包开发环境启动。
- 在项目目录执行：

```bash
OPENCLAW_TEST_MISSING_DEPS=all env -u ELECTRON_RUN_AS_NODE npm run gui
```

### 操作步骤

1. 打开未安装状态首页。
2. 点击“开始准备 OpenClaw”。
3. 观察是否在安装前自动跳转到基础组件准备页。
4. 打开问题排查页面。
5. 查看 Node.js、npm、Git 的诊断状态。
6. 在可控条件下触发安装失败，检查失败后的辅助入口。

### 预期结果

- 首页仍允许点击“开始准备 OpenClaw”。
- 点击后进入首次向导第 1 步。
- 不会因为模拟缺失而在开始前跳到基础组件准备页。
- 问题排查页面将 Node.js、npm、Git 显示为诊断项。
- 安装失败后可进入基础组件建议。
- 模拟只改变 GUI 诊断展示，不修改真实系统环境。

### 备注/异常处理

- 该环境变量仅用于开发测试，正式打包版本必须忽略。
- 如需恢复真实检测，关闭 GUI 后使用正常启动命令重新运行。

## 四、OpenClaw 安装失败流程

### 测试场景

验证官方安装流程失败后的页面、操作入口和流程边界。

### 准备状态

- OpenClaw 未安装。
- 使用可恢复方式模拟安装失败，例如临时断开测试网络或使用项目已有失败模拟方式。
- 不要破坏日常使用环境中的系统组件。

### 操作步骤

1. 从首页点击“开始准备 OpenClaw”。
2. 等待安装流程返回失败。
3. 检查失败页标题、说明和按钮。
4. 点击“重试准备 OpenClaw”。
5. 返回失败页，点击“打开问题排查”。
6. 返回失败页，点击“查看基础组件建议”。

### 预期结果

- 页面显示“准备 OpenClaw 失败”。
- 页面提供：
  - “重试准备 OpenClaw”
  - “打开问题排查”
  - “查看基础组件建议”
- 失败后不进入配置 API Key 步骤。
- 失败页不显示 API Key 表单。
- 重试会再次执行准备流程。
- 问题排查和基础组件建议入口可正常打开。

### 备注/异常处理

- 记录脱敏后的错误摘要，不要在测试记录中粘贴 API Key 或敏感命令。
- 测试结束后恢复网络等临时条件，再验证重试是否可继续。

## 五、OpenClaw 已安装但未配置

### 测试场景

验证存在 OpenClaw 命令，但没有 OpenClaw 配置和 GUI 配置确认状态时的首页与表单校验。

### 准备状态

- 保留可用的 `openclaw` 命令。
- 删除 `~/.openclaw`。
- 删除 `~/.openclaw-installer`。
- 重新启动 GUI。

### 操作步骤

1. 检查首页状态和主按钮。
2. 点击左侧“配置 API Key”。
3. 不填写 API Key，点击“开始配置”。
4. 检查表单是否保留及错误位置。
5. 在 API Key 输入框中开始输入，观察错误是否清除。
6. 选择“自定义模型名称”，保持模型名称为空并提交。
7. 输入自定义模型名称，观察错误是否清除。

### 预期结果

- 首页提示“OpenClaw 已安装，还需要配置 API Key”。
- 主操作为“配置 API Key”，不要求重新安装 OpenClaw。
- 左侧“配置 API Key”进入配置表单。
- API Key 为空时，在输入框附近显示“请先填写 API Key，否则无法完成配置。”
- 表单保持可见，不进入独立错误页，不进入检查配置步骤。
- 用户开始输入 API Key 后，对应错误清除。
- 自定义模型为空时显示“请输入自定义模型名称，或改用自动推荐。”
- 用户输入模型名称后，对应错误清除。
- 无效提交不调用 `runQuickConfigure`。

### 备注/异常处理

- 应使用专用测试 API Key，不要截图、复制或记录真实密钥。
- 可通过非敏感调用记录确认无效提交未调用配置流程，不得记录命令参数。

## 六、OpenClaw 已安装且已配置

### 测试场景

验证 OpenClaw、配置文件、GUI 配置确认状态和检查结果均有效时的工具箱首页。

### 准备状态

- `openclaw` 命令可用。
- `~/.openclaw` 存在且配置有效。
- `~/.openclaw-installer/config-state.json` 存在，且为 GUI 成功配置后生成的非敏感状态。
- 当前配置检查通过。

### 操作步骤

1. 启动 GUI 并等待启动探测完成。
2. 检查首页标题、状态标签和仪表盘。
3. 逐一确认仪表盘卡片。
4. 检查最近操作卡片。
5. 检查页面底部是否出现普通用户不需要的技术命令说明。

### 预期结果

- 首页显示“OpenClaw 已准备好”。
- 不显示首次向导步骤条。
- 完整显示六张仪表盘卡片：
  - OpenClaw
  - 控制台状态
  - API Key
  - 当前模型
  - 配置入口
  - 安全须知
- 最近操作最多显示 3 条有效结果记录，或显示“暂无最近操作”。
- 不显示“已准备好”“正在识别状态”等模糊过程记录。
- 普通页面底部不显示安装助手命令、OpenClaw 命令等技术说明。

### 备注/异常处理

- 如果重启后变为“待确认”，检查 GUI 状态文件是否存在，以及当前配置检查是否通过。
- 不得通过读取或展示 API Key 明文来确认配置。

## 七、重新配置 API Key

### 测试场景

验证已配置用户更换服务商或模型后的配置结果页和控制台跳转。

### 准备状态

- OpenClaw 已安装且已配置。
- 准备专用测试 API Key，不在测试记录中保存或展示。

### 操作步骤

1. 进入“配置 API Key”页面。
2. 更换 provider 或 model。
3. 填写测试 API Key 并开始配置。
4. 等待配置和检查完成。
5. 确认页面显示“配置完成”。
6. 点击“打开控制台”。
7. 观察页面流转和首页状态。

### 预期结果

- 配置成功后显示“配置完成”及控制台操作入口。
- 点击“打开控制台”直接调用现有 `openDashboard`。
- 不进入首次向导第 4 步，不显示首次向导步骤条。
- 启动成功后返回工具箱首页，或在当前结果页显示清晰成功反馈。
- 推荐结果为返回首页，并显示控制台状态更新及“控制台启动成功”最近操作。
- “返回首页”进入已配置用户的工具箱首页。

### 备注/异常处理

- 如启动失败，应留在配置完成页或显示轻量错误，不应进入首次向导。
- 完成测试后撤销或轮换测试 API Key。

## 八、启动和停止控制台

### 测试场景

验证控制台启动、停止、重复启动及操作期间的界面可交互性。

### 准备状态

- OpenClaw 已安装且已配置。
- 首页显示完整仪表盘。

### 操作步骤

1. 在首页点击“启动控制台”。
2. 启动过程中尝试点击左侧导航。
3. 启动完成后检查控制台状态和最近操作。
4. 点击“停止控制台”。
5. 停止后再次点击“启动控制台”。
6. 进入问题排查页，分别测试启动和停止控制台。
7. 停止后检查 `~/.openclaw` 和 `~/.openclaw-installer/config-state.json` 是否仍存在。

### 预期结果

- 启动时只禁用当前启动按钮，不锁住整个 GUI。
- 左侧导航、配置入口和问题排查入口仍可点击。
- 成功后显示控制台运行中，并记录清晰、脱敏的成功操作。
- 停止按钮可点击，成功后显示控制台已停止。
- 停止后可再次启动，界面不会卡在交互提示。
- 问题排查页的启动和停止操作可用。
- 停止控制台不会删除 OpenClaw 配置和安装器配置确认状态。

### 备注/异常处理

- 若停止后再次启动失败，记录中文错误摘要并检查问题排查日志。
- 停止控制台不等于卸载 OpenClaw。

## 九、问题排查页面

### 测试场景

验证诊断中心的常用操作、反馈、检测结果和技术信息。

### 准备状态

- 至少分别在“已安装已配置”和“模拟依赖缺失”状态下执行一次。

### 操作步骤

1. 从左侧导航进入“问题排查”。
2. 点击“重新检查环境”。
3. 点击“检查配置”。
4. 点击“检查更新”。
5. 检查按钮文案、spinner、禁用范围和完成反馈。
6. 测试“查看最近日志”。
7. 检查技术信息区域的默认状态。
8. 展开技术信息，检查内容是否为非敏感信息。
9. 查看 Node.js、npm、Git 的说明和状态。

### 预期结果

- 重新检查环境和检查配置均有立即反馈及结果反馈。
- 检查更新时按钮显示“检查中...”和 spinner。
- 检查更新期间只禁用该按钮，其他页面和导航仍可操作。
- 完成后在问题排查页显示以下结果之一：
  - 已检查更新，当前已是最新版本。
  - 发现新版本，并显示当前版本与最新版本。
  - 暂时无法检查更新，不影响当前使用。
- “查看最近日志”可用。
- 技术信息默认折叠。
- Node.js、npm、Git 作为安装失败后的诊断项显示，不作为普通用户主流程前置。

### 备注/异常处理

- 检查更新失败不应被当作 OpenClaw 安装失败。
- 技术信息及日志不得显示 API Key、API Key 片段或完整敏感命令。

## 十、关于本工具抽屉

### 测试场景

验证关于抽屉的打开、内部更新检查、折叠内容和关闭行为。

### 准备状态

- GUI 位于任意普通页面。
- 夜间模式菜单处于关闭状态。

### 操作步骤

1. 点击左侧“关于本工具”。
2. 检查右侧抽屉标题和分组。
3. 点击“检查更新”。
4. 观察检查中按钮、spinner 和抽屉状态。
5. 等待检查完成并查看抽屉内反馈。
6. 展开和收起“技术信息”。
7. 如发现新版本，点击“稍后再说”。
8. 依次测试 X、Esc 和左侧导航的关闭行为。

### 预期结果

- 右侧打开“关于 OpenClaw 工具箱”抽屉，主页面不跳转。
- 检查更新时显示 spinner 和“检查中...”。
- 点击检查更新及抽屉内普通内容不会关闭抽屉。
- 完成后在抽屉内显示已是最新、发现新版本或失败反馈。
- 技术信息默认折叠，展开/收起不会关闭抽屉。
- “稍后再说”只隐藏本次会话的新版本提示，不关闭抽屉。
- X 和 Esc 可以关闭抽屉。
- 点击左侧导航可以关闭抽屉并进入对应页面。

### 备注/异常处理

- “打开问题排查”允许关闭抽屉并切换页面，这是明确的导航行为。
- 抽屉内不得出现重新检查环境、官方配置向导、启动或停止控制台等维护按钮。

## 十一、夜间模式

### 测试场景

验证外观菜单的默认状态、关闭行为和三种外观模式的可读性。

### 准备状态

- 关闭并重新启动 GUI。
- 可分别准备浅色和深色 macOS 系统外观，用于检查“跟随系统”。

### 操作步骤

1. 启动后观察夜间模式菜单是否可见。
2. 点击左侧“夜间模式”。
3. 依次选择“跟随系统”“浅色模式”“深色模式”。
4. 再次点击夜间模式入口，测试切换关闭。
5. 分别通过点击主内容空白、其他导航和按 Esc 关闭菜单。
6. 在深色模式下检查首次向导步骤卡片。
7. 在深色模式下进入配置 API Key 页面，检查 label、input、select 和 placeholder。
8. 打开关于抽屉，检查 spinner 和反馈文字。

### 预期结果

- 应用启动时菜单默认关闭。
- 点击夜间模式只打开菜单，不切换主内容页面。
- 菜单包含“跟随系统”“浅色模式”“深色模式”。
- 选择后左侧状态文案更新，菜单自动关闭，界面样式立即响应。
- 再次点击、点击空白、点击其他导航或按 Esc 均可关闭菜单。
- 深色背景不是纯黑，卡片和输入框不使用刺眼纯白。
- 深色模式下步骤卡片、表单 label、输入框、下拉框及反馈均清楚可读。
- 浅色模式布局和文字显示正常。

### 备注/异常处理

- 当前会话内状态可接受；如果项目未实现持久化，不要求重启后保持选择。
- 注意检查窄窗口下菜单不要过度遮挡左侧底部状态。

## 十二、左下角状态

### 测试场景

验证左侧边栏底部状态文案与实际工具箱状态一致。

### 准备状态

依次准备以下状态：

1. OpenClaw 未安装。
2. OpenClaw 已安装但未配置。
3. OpenClaw 已安装且已配置，控制台未运行。
4. 控制台运行中。
5. 存在安装、配置或控制台异常。

### 操作步骤

1. 在每种准备状态下启动或刷新 GUI。
2. 等待状态探测完成。
3. 观察左下角状态文案和右侧版本号。
4. 在检查过程中观察临时状态。

### 预期结果

| 实际状态 | 左下角文案 |
|---|---|
| 正在检测 | 检测中 |
| OpenClaw 未安装 | 待准备 |
| 已安装但未配置 | 待配置 |
| 已安装且已配置，控制台未运行 | 工具箱正常 |
| 控制台运行中 | 控制台运行中 |
| 存在需处理异常 | 需要处理 |

- 右侧显示 `v1.0.0`。

## 十三、安全检查

### 测试场景

验证 API Key、命令、状态文件、日志和最近操作的敏感信息边界。

### 准备状态

- 准备专用测试 API Key。
- 打开配置页面、最近操作、问题排查、关于抽屉和相关日志。
- 不使用生产环境 API Key。

### 操作步骤

1. 在 API Key 输入框中输入测试值并完成一次配置。
2. 检查 GUI 输出、错误提示和最近操作。
3. 关闭并重启 GUI，检查 API Key 输入框。
4. 检查 `~/.openclaw-installer/config-state.json`。
5. 检查安装器最近日志和诊断信息。
6. 检查关于抽屉及问题排查技术信息。

### 预期结果

- GUI 不保存或回填 API Key。
- 除用户正在输入的密码输入框外，不显示 API Key 明文或片段。
- 最近操作不记录 API Key、完整 onboarding 命令或敏感路径。
- `config-state.json` 只包含 provider、model、配置时间、版本等非敏感状态，不包含 API Key 或可还原片段。
- 安装器日志不记录带 API Key 的完整命令。
- 技术信息不展示 API Key 或含密钥日志。
- 表单校验和错误摘要不会回显用户输入的 API Key。

### 备注/异常处理

- 如果发现任何密钥泄露，立即停止测试、撤销测试 Key，并将问题标记为发布阻塞。
- 测试记录只描述泄露位置，不粘贴实际密钥内容。
